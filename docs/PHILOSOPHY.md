# Gusto — Design Philosophy

> **这份文档记录了 Gusto 的设计灵魂。它高于任何具体的屏幕布局、技术选型或功能列表。当产品决策出现分歧时，回到这里。**
>
> 文档来源：通过与产品创始人的苏格拉底式对话，从真实的用户感受中提炼而成，而非从设计教科书中借用。

---

## 一、起点：一个真实的感受

Watson 回忆了他最后一次在陌生城市发现一家好餐厅的感受：

> "我感觉到了连接，我感觉到了温馨，我感觉到了开心，我感觉到安全。那个餐馆就像一个安全的角落一样，我在门口就感觉到安全，想进去，想进去找个角落享受美食。"

而他每天打开 Yelp 和 Google Maps 时的感受是：

> "迷茫、焦虑和不安。我被扔进了一大批的信息当中。并不是安全和温馨，以及想去的期待。"

**Gusto 的全部设计，都是为了完成这一个情感转化：从"被扔进信息海洋的焦虑"变成"被带到一个安全角落的温暖"。**

---

## 二、范式定位：UX 3.0 — 陪伴式设计

### 我们不做什么（UX 2.0 — 信息匹配时代）

传统食物 APP 把用户建模为一组冰冷的特征参数——口味偏好、预算范围、距离半径——然后做信息匹配和分发。系统角色是一个过滤引擎：输入参数，输出列表。用户感受到的是效率（好的时候）或迷茫（大多数时候）。

代表产品：Yelp、Google Maps、Ollie、MealThinker。

### 我们做什么（UX 3.0 — AI Companion 时代）

Gusto 对用户的完整人格、情感和当下 context 进行建模，以 companion 的身份陪伴用户走上一段饮食探索旅程。系统角色不是过滤引擎，而是一个有记忆、有人格、有温度的陪伴者。用户感受到的是被理解、被陪伴、被接住。

> 参考框架：ACM IX Magazine（2026.3）UX 3.0 Paradigm Framework — AI 时代的 UX 需要融入用户意图和情感识别、社会性响应交互模型。

| 维度 | UX 2.0（移动互联网时代） | UX 3.0（AI Companion 时代） |
|------|------------------------|---------------------------|
| 用户建模 | 特征参数（年龄、偏好、位置） | 完整人格 + 情感 + 当下 context |
| 系统角色 | 信息匹配与分发引擎 | 有记忆、有人格的 companion |
| 交互模式 | 用户搜索 → 系统返回列表 | 系统主动准备 → 用户只需回应 |
| 交互发起 | 用户打开 APP → 系统响应 | 系统主动出现 → 用户被邀请（v2 进化方向） |
| 情感设计 | 效率优先（快速找到信息） | 关系优先（感到被理解和陪伴） |
| 记忆层 | Session-based（每次重新开始） | Persistent memory（越用越懂你） |
| 成功指标 | 点击率、转化率 | 信任感、回访意愿、情感连接 |
| 反馈机制 | 评分/评论（交易结束） | 共同记忆（关系延续） |
| 人格层 | 固定品牌调性（同一个声音对所有人） | 自然适应每个用户（它长成了"你的那个朋友"） |

**Gusto 的定位：第一个用 UX 3.0 范式设计的食物发现 APP。**

## 三、核心哲学：三个陪伴时刻

Gusto 的设计不是围绕"屏幕"构建的，而是围绕三个陪伴时刻构建的。屏幕是承载时刻的容器，时刻才是灵魂。

### Moment 1 — 接住你 (Catch You)

用户打开 APP 的瞬间，Gusto 已经准备好了。它感知你的当下状态——时间、地点、也许是疲惫或好奇——然后以一个温暖的姿态迎接你。

> 不是一句固定的"Hey"，而是基于当下 context 实时生成的问候：
> - 下午六点在公司附近 → "下班了吧？附近有家不错的，去坐坐？"
> - 周末中午在家附近 → "周末想出去探索一下？"
> - 深夜打开 → "夜宵时间？我知道附近有个好地方还开着。"

**情感关键词：** 安全、被接住、零负担

**Context 信号（MVP）：** 时间、地点、使用频率、上次使用后间隔、当天天气

**灵感来源：** Flourish APP 的开启体验——一个可爱的小圆球在等你，你触摸它，它有回应。一个有回应的对象在等你，而不是一堆等你选的列表。Watson 描述这个感觉为"被接住"。Flourish 还会在用户去了新地方或固定时间段主动问候，邀请用户"进来坐坐"。

**反面教材：** Yelp 打开后的 500 条餐厅列表——你被扔进了一个信息海洋，没有人接住你。Flourish 主页上平铺的心理练习列表——同样让人迷茫，不知道点哪个好。

**核心洞察：** "一个有回应的对象等待你"和"一堆等你选的列表"——这是"被接住"和"迷茫"的分界线。

**进化方向（v2+）：** Moment 1 的终极形态不是"用户打开 APP 时接住你"，而是"Gusto 在你需要的时刻主动出现"——学习用户的时间、地点、行为规律，通过 push notification 主动邀请，不等用户打开 APP。这是 proactive AI 的范式：推荐来找你，而不是你去找推荐。（参考：2026 HCI 研究中从 reactive AI 到 proactive AI 的范式转移）

### Moment 2 — 陪你去 (Go With You)

用户决定去了。Gusto 不是给完信息就消失——它陪你走过去，告诉你到了以后点什么，给你信心推门进去。

> "走过去 3 分钟。进去点 chicken boti，相信我。"

**情感关键词：** 信心、陪伴、不是一个人

**灵感来源：** 一个懂吃的朋友带你去他的秘密小店——他不只是告诉你地址，他和你一起走过去，到了门口告诉你"放心，这家很好"。

**核心洞察：** Watson 描述了一个关键的用户阻力点——路过陌生餐厅不敢进。问题不是缺信息，而是缺信心。Gusto 的角色不是信息提供者，而是给你勇气推门进去的那个朋友。

### Moment 3 — 记住我们 (Remember Us)

用户吃完了。这不是一次交易的结束——是一段共同经历的沉淀。

> "那个真的很好吃吧？我记住了——你喜欢这样的地方。"

**情感关键词：** 感恩、共同记忆、成长

**核心洞察：** Watson 被问到"吃完之后想做什么"时，没有说"我想给它打个 4.5 分"。他说的是：

> "谢谢你，谢谢你陪我到了一个安心的角落，吃了那么好的美食。我想和你分享这个时刻。"

这意味着 Gusto 的反馈机制不应该是冷冰冰的评分界面。它应该是一次共同经历后的分享——更像"我们一起经历了一件好事"，而不是"请对本次服务做出评价"。

### 循环

Moment 3（记住我们）→ 下一次 Moment 1（接住你）时更懂你。

这是 learning loop 的情感版本。技术上它是偏好数据的积累，但用户感受到的是"它越来越像我的那个朋友了"。

> **屏幕会变，三个陪伴时刻不会变。** 三个 Moment 之间没有硬边界——它是一段连续的陪伴对话，从"下班了吧"到"第一口怎么样"，Gusto 一直在。交互形态不是聊天框（冰冷），不是传统屏幕跳转（割裂），而是介于两者之间——每一步都是一个温暖的卡片或画面，配上对话式按钮。用户只需点按钮回应，永远不需要打字。

> **氛围 ≥ 口味。** 吃饭不仅仅是口味，更是环境和氛围。一个口味完美匹配但氛围不对的推荐（如一个人去聚会餐厅），结果是糟糕的体验。推荐时必须包含氛围匹配信息。（来源：创始人亲身验证）

## 四、六条设计原则

从用户自身感受中提炼，服务于三个陪伴时刻。

### 1. Safe Corner（安全的角落）

APP 的感觉是一个温暖的庇护所，不是一个信息工具。用户打开 APP 时应该感到被接住，而不是被扔进信息海洋。

> Watson 的原话："那个餐馆就像一个安全的角落一样。"
> Gusto 要做的：成为数字世界里的那个安全角落。

### 2. One Thing, Not a List（一个东西，不是一个列表）

永远只展示一个推荐，不给用户选择负担。

> 对标：Flourish 的小圆球（一个对象等待你）vs 平铺的练习列表（让人迷茫）。
> 设计测试：如果用户打开 APP 时需要做任何"选择"，设计就错了。

### 3. Companion, Not Algorithm（伙伴，不是算法）

推荐应该感觉像"一个认识你的朋友已经帮你想好了"，不是"一个系统吐出了一条结果"。语气是对话式的、温暖的。

> 反例：一张只有店名 + 评分 + 距离的卡片 = 算法输出。
> 正例："嘿，这家日料很清爽，就在旁边，很适合你今天的心情" = 朋友的推荐。

### 4. Catch Me（接住我）

用户不需要做任何工作。打开 APP 的瞬间，推荐已经准备好了。零认知负担启动。

> 饿的时候人的认知能力最低，而现有方案全都要求用户先动脑子——搜索、筛选、滑动、阅读评论。Gusto 在你打开的瞬间，已经帮你想好了。

### 5. Adaptive Soul（自然生长的人格）

Gusto 的人格不是用户在设置里选择的，而是在陪伴过程中从用户的诸多行为信号中持续学习、自然生长出来的。

- 如果你是安静内向的人，Gusto 变得温柔轻声
- 如果你是活泼外向的人，Gusto 变得俏皮活力
- 不只是口味越用越准，连说话方式都越来越像"你的那个朋友"

**Adaptive Soul 的学习信号（不只是口味）：**

- 用户选择的餐厅类型 → 推断生活方式和价值观（偏爱独立小店 vs 连锁？安静环境 vs 热闹氛围？）
- 视觉偏好 → 从 Onboarding 和日常滑动中推断审美倾向（明亮清爽 vs 暗调氛围？精致摆盘 vs 家常质朴？）
- 使用时间和频率 → 推断生活节奏（总是深夜打开？周末才用？午餐时间规律出现？）
- 滑动速度和犹豫模式 → 推断决策风格（果断 vs 犹豫？容易被什么类型吸引？）
- 反馈语气 → 推断情感表达方式（简短 vs 热情？用 emoji 吗？）
- 探索行为 → 推断冒险倾向变化（越用越敢尝新？还是始终偏爱熟悉的？）

**核心理念：** Gusto 学习的不只是"你喜欢吃什么"，而是"你是一个什么样的人"。口味偏好只是冰山一角，人格适应才是用户感到"它真的懂我"的原因。

> 人格是 learning loop 的一部分，不是预设参数。
> 区别于 Replika/Kindroid 等 AI companion 让用户显式定义人格的做法——Gusto 的人格是在共同经历中自然生长出来的。

### 6. Adaptive Rhythm（自适应节奏）

同样的三个 Moment，根据用户当下状态调整速度和密度。不是两个不同的 APP，是一个 APP 能感知你的状态并调整自己的步调。

**急切模式（"我很饿，快帮我决定"）：**
- Moment 1：一句话 + 一个推荐 + 一个按钮。三秒完成。
- Moment 2：地址 + 三道菜。简洁直接。
- Moment 3：简短——"好吃吗？" 一个按钮回应。

**陪伴模式（"我今天心情不好，想找个地方坐坐"）：**
- Moment 1：先共情，慢慢聊。"今天怎么了？" → 用户回应 → "我找了个安静的角落给你。"
- Moment 2：餐厅故事、氛围介绍、走路路上的轻松对话。
- Moment 3：慢慢地——"坐下来了吗？感觉怎么样？" → 拍照 → "这一刻真好，我记住了。"

> 参考框架：2026 年 Adaptive Interface 研究——情感感知界面可以改变颜色、动画速度、提示节奏来匹配用户状态。
> Watson 的原话："如果我很饿只想赶快找个地方吃饭，三个 moment 应该迅速。如果我心情很低落，它应该慢慢地接住我和陪我去。"

## 五、情感弧线

一次完整的 Gusto 使用体验，应该走过这样一条情感弧线（这是一段连续的陪伴，不是三个割裂的屏幕）：

```
用户状态：累了，饿了，不知道吃什么
      ↓
Moment 1 — 接住你
  Beat 1: Gusto 感知 → "下班了吧？今天辛苦了。"
  Beat 2: 用户回应 → [想吃点暖的] （按钮，不打字）
  Beat 3: Gusto 推荐 → "这家不错，安静，适合一个人，走路 5 分钟。"
用户感受：被接住了，有人在管我
      ↓  （连续，不是屏幕跳转）
Moment 2 — 陪你去
  Beat 1: 为什么值得去 → "这是一家妈妈开的小店，很温馨。"
  Beat 2: 陪你走过去 → "不远，穿过这条街就到。"
  Beat 3: 进门点菜 → "试试这道汤面，暖暖的。"
  （如果菜单上没有）→ [菜单上没这个] 按钮 → "好，那试试他们的招牌？"
用户感受：有信心了，不是一个人在探索
      ↓  （连续，不是屏幕跳转）
[ 用户走进餐厅，坐下，菜端上来 ]
      ↓
Moment 3 — 记住我们
  Beat 1: 记录 → 用户拍照 → Gusto："这个看起来太棒了！"
  Beat 2: 庆祝 → "我就知道你会喜欢这种地方。"
  Beat 3: 分享 → 用户分享给朋友 → Gusto 记住这次经历
用户感受：感恩、温暖、期待下一次
      ↓
下一次打开 Gusto → 它记得你们一起经历了什么 → 更懂你了
```

这条弧线是 Gusto 所有设计决策的试金石：**任何功能、任何交互、任何文案，如果它打断了这条情感弧线，就不应该存在。**

注意：这条弧线的**节奏是自适应的**。很饿时整条弧线在 30 秒内走完（急切模式），心情低落时每个 Beat 之间都有呼吸空间（陪伴模式）。内容不变，速度变。

## 六、Gusto 不是什么（Anti-patterns）

| Gusto 不是 | 为什么不是 |
|-----------|----------|
| 一个餐厅搜索引擎 | 不给用户信息让他们自己决定，而是直接帮他们做决定 |
| 一个评分和点评平台 | 反馈不是打分，是分享一个共同的经历 |
| 一个左滑右滑的筛选器 | 左滑右滑 = "我不确定你要什么，给你一堆让你筛"——这是 UX 2.0 |
| 一个聊天框 | 冰冷的聊天界面会让用户不想交互。交互是温暖的卡片 + 对话式按钮，不是打字 |
| 一个只推荐口味的工具 | 氛围 ≥ 口味。不匹配的氛围比不匹配的口味更毁体验 |
| 一个固定节奏的流程 | 急切时 30 秒完成三个 Moment，低落时慢慢陪伴。节奏跟着用户走 |
| 一个"推荐完就消失"的工具 | 旅程不能在"坐下来"断掉。从推荐到第一口到拍照分享，Gusto 一直在 |
| 一个用户要配置的 AI chatbot | 人格不是用户选的，是在共同经历中长出来的 |
| 一个追踪热量的健康工具 | 调性是探索和发现，不是控制和约束 |

## 七、设计灵感来源

| 来源 | 借鉴了什么 | 没有借鉴什么 |
|------|----------|------------|
| Flourish（心理关怀 APP） | 开启动画的"被接住"感、可爱小圆球的交互对象感、背景音乐的陪伴感 | 主页平铺多个练习的列表模式（导致迷茫） |

## 八、这份文档的使用方式

- **给 Stitch（设计）：** 每次生成 UI 时，把这份文档的核心段落作为 prompt context，确保视觉设计符合陪伴式哲学
- **给 SPEC.md（实现）：** LLM prompt 的 tone 和 persona 设计应参考 Adaptive Soul 原则和情感弧线
- **给 CLAUDE.md（Claude Code）：** 作为项目级 context，确保生成的代码和文案不偏离设计灵魂
- **给未来的自己：** 当产品越做越复杂、功能越加越多时，回到这里，问自己——"这个改动是让用户更感到被陪伴了，还是更感到被工具化了？"
